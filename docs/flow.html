📖 The MyMeal Story

What are we building?

MyMeal is a bridge between the comfort of home-cooked food and the fast-paced life of modern consumers. We are building a hyper-local meal subscription engine that empowers home chefs ("Cooks") to run their own professional kitchens while providing "Consumers" with healthy, predictable, and customizable meal plans.

Unlike traditional food delivery, MyMeal focuses on long-term relationships via subscriptions, ensuring cooks have stable income and consumers have one less daily chore to worry about.

How are we building it?

The backend is engineered for reliability and transparency using the following principles:

  • Robust Foundation: Powered by AdonisJS v6 for its elegant TypeScript-first architecture and developer-friendly abstractions.
  • Data Integrity: Every meal plan creation and subscription event is wrapped in SQL Transactions to prevent data corruption.
  • Daily Order Engine: Instead of static schedules, we use a dynamic OrderService that generates individual orders for every valid subscription day, allowing for granular status tracking.
  • Real-time Trust: Integrated with WhatsApp Web.js to send instant notifications for every payment, earning, and order note, keeping both parties in sync without checking the app constantly.
  • Financial Ledger: A dual-entry style ledger tracks amountPaid vs amountConsumed, allowing for flexible payouts and advance payments.

🔗 Middleware Pipeline

Middleware runs in 3 layers. Server middleware runs on every request. Router middleware runs on matched routes. Named middleware is explicitly attached to specific routes.

Layer 1: Server Middleware (Every Request)

ForceJsonResponse→ ContainerBindings→ CORS
MiddlewareFileWhat It Does
ForceJsonResponseforce_json_response_middleware.tsSets Accept: application/json header on every request so AdonisJS always returns JSON (not HTML error pages)
ContainerBindingscontainer_bindings_middleware.tsBinds HttpContext and Logger to the IoC container resolver for each request — enables dependency injection
CORS@adonisjs/corsHandles Cross-Origin Resource Sharing headers so the frontend can call the API from a different domain

Layer 2: Router Middleware (Matched Routes Only)

BodyParser→ Session→ Shield (CSRF)→ InitializeAuth→ SilentAuth
MiddlewareWhat It Does
BodyParserParses JSON, form data, and multipart request bodies into request.body()
SessionInitializes session store (cookie-based)
ShieldCSRF protection for form submissions
InitializeAuthBootstraps the AdonisJS auth system, makes ctx.auth available
SilentAuthSilently checks if a valid token is present. Does NOT block the request if unauthenticated — just sets ctx.auth.user if token is valid

Layer 3: Named Middleware (Route-Specific)

auth()→ role('cook' | 'consumer')
MiddlewareFileWhat It Does
AuthMiddlewareauth_middleware.tsCalls ctx.auth.authenticateUsing() — verifies the Bearer token in the Authorization header. Returns 401 Unauthorized if token is missing/invalid/expired
RoleMiddlewarerole_middleware.tsChecks auth.user.role against allowed roles. Supports string ('cook'), array, or object format. Returns 403 Forbidden if role mismatch

Exception Handler

File: app/exceptions/handler.ts

Extends AdonisJS ExceptionHandler. In development mode (!app.inProduction), shows verbose errors with stack traces. In production, returns clean error responses. Catches validation errors, auth errors, and unhandled exceptions.

🔐 Auth & Profile APIs

POST /api/v1/auth/register

Public

Middleware Chain

Server MW→Router MW→No named MW (public)

Validator: signupValidator

FieldRules
namestring, required
emailstring, email, max 254, unique in users table
phonestring, unique in users table
passwordstring, min 8, max 32
passwordConfirmationstring, must match password
roleenum: 'consumer' | 'cook'

Controller Flow

1
Validate — request body is validated against signupValidator. If email/phone already exists in DB, VineJS throws 422.
2
Hash Password — hash.make(payload.password) uses scrypt/argon2 to hash the password.
3
Create User — User.create() inserts into users table with hashed password and default role.
4
Generate Token — User.accessTokens.create(user) creates an opaque access token in auth_access_tokens table.
5
Transform & Respond — UserTransformer strips password, returns user + token. Status 201.

}

POST /api/v1/subscriptions

🔒 auth + role(consumer)

Validator: createSubscriptionValidator

FieldRules
mealPlanIdstring, required
pickupSlotIdstring, required
startDatestring (ISO date), required
durationenum: one_time|1_week|2_week|1_month|3_month, optional
advancePaymentnumber, min 0, optional
components[]array min 1: { mealComponentId, quantity, enabled? }

Controller Flow (Most Complex API)

1
Validate — createSubscriptionValidator
2
Check Meal Plan — Must exist AND be active. Returns 404 otherwise.
3
Check Capacity — Counts active/paused subs for this plan. If ≥ subscriberLimit → 400.
4
Validate Pickup Slot — Must belong to this meal plan.
5
Calculate Date Range — calculateSubscriptionRange(startDate, duration, validityType) → endDate + validDays count (skips weekends/weekdays per validityType)
6
Price Components — For each component: if toggle → add price when enabled; if quantity → add (qty - defaultQty) × price for extras. Sum = dailyExtraPrice.
7
Total = (basePrice + dailyExtraPrice) × validDays
8
Advance Check — advancePayment must be ≥ 10% of totalPrice. Returns 400 with minAdvance if too low.
9
Transaction — Creates Subscription + SubscribedMealComponents atomically
10
OrderService.generateOrders() — Auto-generates one Order per valid day (status: 'scheduled')
11
PaymentService.processPayment() — Records advance payment, updates amountPaid on subscription

PATCH /api/v1/orders/:id/status

🔒 auth + role(cook)

Body

{ "status": "prepared" | "picked_up" }

Controller Flow

1
Validate Status — Only 'prepared' and 'picked_up' are accepted. 400 otherwise.
2
Find Order — Preloads subscription→mealPlan
3
Authorization — Checks that auth.user's cookProfile.id matches order.subscription.mealPlan.cookId. 403 if not.
4a
If 'prepared' — Simply updates order.status to 'prepared' and saves.
4b
If 'picked_up' — Calls OrderService.syncFinancials(order, 'picked_up') which:
• Sets order status to 'picked_up'
• Adds order.price to subscription.amountConsumed

⚙️ Services Deep Dive

Services encapsulate business logic, keeping controllers thin. They handle transactions, financial calculations, external integrations, and notifications.

📦 OrderService

app/services/order_service.ts

The brain of order lifecycle management. Handles order generation, pause/resume, edit, financial sync, and auto-miss logic.

MethodCalled ByWhat It Does
generateOrders(sub)SubscriptionsController.storeLoops from startDate → endDate, creates one Order per valid day (respects validityType: weekdays/weekends/all_days). Each order gets status='scheduled', price=dailyPrice.
handleSubscriptionPause(sub)SubscriptionsController.updateStatusDeletes all future orders with status='scheduled' and orderDate ≥ today.
handleSubscriptionResume(sub)SubscriptionsController.updateStatusRe-generates orders from today → endDate, checks for existing orders to avoid duplicates.
handleSubscriptionEdit(sub)SubscriptionsController.updateDeletes future scheduled orders, then calls handleSubscriptionResume to re-generate with updated pricing.
syncFinancials(order, status)OrdersController.updateStatusTransaction: Sets order status → adds order.price to subscription.amountConsumed → calls PaymentService.updateCookWallet → checks if subscription is done (no more scheduled orders) → marks 'completed'
handleAutoMissed()Cron job (external)Finds 'prepared' orders where pickupTime + 3 hours has passed. Calls syncFinancials(order, 'missed') for each.

getDailyPrice() Logic

dailyPrice = mealPlan.basePrice + SUM(
  for each subscribedMealComponent where enabled:
    component.price × component.quantity
)
